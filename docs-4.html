Introduction

Klar is a lightweight visual CMS for static websites. Point at any live webpage, click on text, images, links, or icons to edit them inline, and save the changes back to a GitHub repository.

Click-to-Edit
Hover to highlight, click to activate inline editing on any text, image, SVG, or link element.
Clean HTML Output
Edits are applied server-side with Cheerio. No injected wrappers, no data attributes in your output.
GitHub-Native
Commit edits directly to your repo, publish to GitHub Pages, and manage media files through GitHub's API.

Features#

  • Click-to-edit — hover to highlight, click to activate inline editing on any text element
  • Structured panel — edit text fields, attributes (href, src, alt, class, style), and raw HTML in a sidebar form
  • Image and media management — upload, browse, and swap images via the GitHub-backed media library
  • Icon picker — search and insert icons from the Iconify catalog
  • SVG editing — replace SVG markup directly through the structured panel
  • Link editing — edit href on anchor elements, even when the link wraps images or mixed content
  • Section navigation — sidebar lists page sections with scroll-to support
  • Script toggle — enable or disable JavaScript in the editor iframe
  • Save and publish — commit edits to GitHub and optionally publish to GitHub Pages
  • Google SSO — sign in with Google to persist configuration across devices

How it Works#

Klar renders your target page in a sandboxed iframe via srcdoc. When you click on any element, it becomes editable. Changes are tracked by CSS selector using a centralized useReducer and applied server-side using Cheerio before being committed back to your GitHub repository.

HTML 
<!-- Your original HTML -->
<section class="hero">
  <h1>Welcome to My Site</h1>
  <p>This is the description.</p>
</section>

<!-- After editing with Klar — still clean! -->
<section class="hero">
  <h1>Welcome to My Updated Site</h1>
  <p>A better description here.</p>
</section>

Tech Stack#

Layer Technology
Frontend React 18, TypeScript, Vite 6, Tailwind CSS 4
Backend Express 4 (Node.js, plain JS), Cheerio
Database SQLite via better-sqlite3 (users, sessions, projects)
Auth Google Identity Services, cookie-based sessions
External APIs GitHub REST API v3, Iconify Design API

Two Modes#

  • Logged out — edit in read-only mode (no save). Enter a website URL and file path to start editing locally.
  • Logged in — full GitHub integration with save, publish, media management, and project persistence via the dashboard.

Installation

Get Klar up and running locally with Node.js.

Prerequisites#

  • Node.js 18.11+ (uses node --watch for auto-reload)
  • A GitHub personal access token with repo and pages scope (for save/publish)
  • A Google Cloud project with OAuth 2.0 Client ID (optional, for SSO)

Install#

Terminal 
# Clone the repository
git clone <repo-url>
cd klar

# Install dependencies
npm install

Environment Setup#

Create a .env file in the project root:

.env 
GOOGLE_CLIENT_ID=<your Google OAuth client ID>
VITE_GOOGLE_CLIENT_ID=<same value>
KLAR_ENCRYPTION_KEY=<32-byte hex string for AES-256-GCM>

Generate an encryption key:

Terminal 
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
Important: Never commit your .env file to version control. The encryption key protects GitHub tokens stored in the database.

Commands#

Command Purpose
npm run dev Start Vite (port 5173) + Express (port 3001) concurrently with auto-reload
npm run build TypeScript check + Vite production build
npm start Production mode (serves dist/ from Express)
npm run typecheck TypeScript type checking only
npm run server Backend only with --watch
In development, Vite proxies /api requests to the Express server automatically. In production, Express serves the built frontend from dist/.

Quick Start

Go from zero to editing your first page.

1. Start the Dev Server#

Terminal 
npm run dev

This starts both the Vite dev server on http://localhost:5173 and the Express backend on http://localhost:3001. Vite proxies /api requests to Express automatically.

2. Configure Your Project#

Open http://localhost:5173 in your browser. You'll see the setup form. You can either:

  • Without login — enter a website URL to fetch and edit HTML locally (no save)
  • With Google SSO — sign in to access the dashboard, create projects, and save/publish to GitHub

For a logged-in project, enter your GitHub repository details:

  • Website URL — the live URL of your site (used to fetch HTML and resolve relative paths)
  • GitHub repo — in owner/repo format
  • File path — path to the HTML file in the repo (e.g., index.html)
  • GitHub token — personal access token with repo scope (encrypted with AES-256-GCM in the database)

3. Start Editing#

The app loads your page in a sandboxed iframe. Hover over any element to see it highlighted, then click to activate editing. Text elements become contentEditable, while images and SVGs open a structured panel for attribute editing.

Press Enter to confirm a text edit, Escape to revert. Use Cmd+S to save or Cmd+P to publish.

4. Save to GitHub#

Click the Save button in the toolbar. Klar fetches the latest file from GitHub, applies your edits server-side using Cheerio, and commits the result. Your original HTML structure is preserved — no wrapper divs or data attributes are added.

Visual Editing

How click-to-edit works in the sandboxed iframe.

Editable Elements#

Klar automatically detects editable leaf elements based on tag name. The following HTML tags are editable by default:

h1 h2 h3 h4 h5 h6 p span a button li td th label blockquote figcaption dt dd em strong small cite summary time

Additionally, <img> and <svg> elements are detected for attribute/markup editing.

Three Edit Types#

Klar supports three types of edits, each applied differently by Cheerio on the server:

Type Description Example
text Updates the text content of an element Changing a heading or paragraph
attribute Modifies an HTML attribute value Changing an image src, link href, or element class
outerHTML Replaces the entire element markup Swapping an SVG icon or restructuring a component

CSS Selector Identity#

Elements are identified by CSS selectors throughout the entire stack — from iframe detection through React state to server-side Cheerio application. The selector strategy uses #id when available and unique, falling back to tagName:nth-of-type(n) chains.

Edit state is centralized in a useReducer keyed by CSS selector (optionally suffixed with ::attribute for attribute edits). The reducer preserves the first original value so reverts always go back to the true original, regardless of intermediate edits.

Naked Text Wrapping#

Mixed-content elements (containing both child elements and bare text nodes) are problematic for contentEditable. Klar solves this by wrapping bare text in <span data-klar-wrap> before editing and unwrapping after save. This happens transparently — the wrapping spans never appear in your saved HTML.

Container Editing#

Elements with repeating children (e.g., a <ul> with multiple <li> items) can be edited as containers. Klar detects these via data-klar-list attributes or custom container selectors, and provides an "Edit list" button that lets you add, remove, reorder, and duplicate items.

Structured Panel

The sidebar form for editing text, attributes, and HTML alongside the visual editor.

Overview#

The structured panel (StructuredPanel.tsx) provides a form-based interface for precise editing. When you click an element in the iframe, the panel shows:

  • Text content — editable text field with live preview sync
  • Attributes — href, src, alt, class, style, and other attributes
  • Raw HTML — full outerHTML editing for advanced use cases

Sections Panel#

The left sidebar (SectionsPanel.tsx) lists page sections detected from semantic HTML elements like <header>, <main>, <section>, <footer>. Clicking a section scrolls the iframe to that element. Custom section selectors can be configured per project.

Element Discovery#

On page load, the iframe script walks the DOM and discovers all editable elements, sending them to the parent via klar-elements and klar-sections messages. This populates both the sections panel and enables search/filtering across all page content.

Iframe Communication

How the parent React app and the sandboxed iframe exchange data via postMessage.

Sandboxed Iframe#

The visual editor renders target HTML in a sandboxed iframe using the srcdoc attribute. Before injection, the HTML is sanitized: scripts are optionally removed, event handlers are neutralized, and a <base> tag is injected for relative URL resolution.

A custom script (iframeScript.ts) is injected into the iframe to handle click detection, hover highlighting, content editing, and element discovery.

Message Protocol#

Parent and iframe communicate via window.postMessage. Each message has a type field that determines its shape. The parent uses a sendToIframeRef held by EditorLayout.

Parent → Iframe

Type Purpose
klar-update-text Update text content of an element by selector
klar-update-attr Update an attribute value on an element
klar-replace-html Replace entire outerHTML of an element
klar-replace-container Replace innerHTML of a list container
klar-scroll Scroll to an element by selector
klar-highlight Highlight an element temporarily
klar-revert Revert text content to original
klar-select-element Programmatically select an element
klar-get-html Request the current clean HTML from the iframe
klar-set-ai-mode Toggle AI editing mode (any element selectable)
klar-update-css Update a custom CSS stylesheet in the iframe
klar-update-js Update a custom JS script in the iframe

Iframe → Parent

Type Purpose
klar-activate User clicked an editable element (includes selector, tagName, text, attributes)
klar-deactivate Active element was deselected
klar-edit Text content was changed (includes selector, original, edited)
klar-elements Full list of discovered editable elements on the page
klar-sections Page sections for the sections panel
klar-activate-container User activated a list container (includes items)
klar-container-updated Container contents changed after an edit
klar-html Response to klar-get-html with clean HTML
klar-hotkey User pressed Cmd+S or Cmd+P in the iframe

Media & Icons

Upload, browse, and swap images via the GitHub-backed media library, and search icons from Iconify.

Media Library#

When you click on an image element, the structured panel shows the current src and alt attributes, plus a button to open the media library dialog (MediaDialog.tsx). The dialog lets you:

  • Browse existing media files from the configured media folder in your GitHub repo
  • Upload new images directly to the repo via the GitHub API
  • Delete unused media files
  • Select an image to update the src attribute

Media files are stored in the configurable mediaFolder path within your repository (defaults to an images/ or assets/ folder).

Icon Picker#

For SVG icons, Klar integrates with the Iconify Design API. The icon dialog (IconDialog.tsx) lets you search across thousands of icon sets, preview icons at different sizes, and insert them as inline SVG — replacing the existing SVG element's markup via an outerHTML edit.

Background Images#

Elements with inline background-image styles can be edited using the :style(background-image) selector modifier. This allows swapping background images through the media library just like regular <img> elements.

GitHub Integration

How Klar uses the GitHub REST API v3 for file operations, caching, and publishing.

ETag-Based Caching#

Both file and media API calls use HTTP ETag caching to minimize GitHub API usage. The pattern:

  1. Store response + ETag in an in-memory Map
  2. On subsequent requests, send If-None-Match header
  3. On 304 Not Modified, return cached data

The file cache is manually updated after successful commits, so subsequent reads don't require an API call. This keeps Klar well within GitHub's rate limits even for active editing sessions.

Server-Side HTML Manipulation#

All HTML transforms use Cheerio server-side in server/github.js:

  • Sanitization — remove scripts, neutralize event handlers, inject <base> tag
  • Naked text wrapping — wrap bare text nodes for contentEditable compatibility
  • Edit application — apply selector-based edits (text, attribute, outerHTML)
  • Unwrapping — remove data-klar-wrap spans before saving
Sanitization only affects the editor preview. When saving, edits are applied to the original unsanitized HTML from your repository, so the output is always clean.

Page Management#

Beyond editing a single file, Klar supports multi-page operations via the GitHub API:

  • Duplicate page — copy an HTML file within the repo
  • Delete page — remove an HTML file from the repo
  • Rename page — move/rename an HTML file
  • List HTML files — browse available pages in the repo
  • Sync sections — share header/footer sections across multiple pages

Save & Publish

Commit your edits to GitHub and publish to GitHub Pages.

Save Flow#

When you click Save (Cmd+S), Klar:

  1. Fetches the latest file content from GitHub (with ETag caching)
  2. Loads the HTML into Cheerio and wraps naked text nodes
  3. Applies each edit by CSS selector — text content, attributes, or outerHTML replacement
  4. Unwraps data-klar-wrap spans
  5. Commits the modified file back to the configured branch
  6. Updates the local file cache with the new content and SHA

Publish Flow#

The Publish action (Cmd+P) goes further:

  1. Saves changes to the main branch (same as Save)
  2. Creates or updates a klar-publish branch from the latest commit
  3. Enables GitHub Pages on the klar-publish branch
  4. Returns the Pages URL for instant preview
There is also a "Publish latest" option that publishes the current state of the main branch to GitHub Pages without saving new edits first. This is useful when you've already committed changes and just want to deploy.

Commit History#

Klar can fetch commit history for the current file via /api/github/commits and retrieve file content at any past commit via /api/github/file-at-commit. This enables browsing previous versions of your page.

Project Settings

Configure each project with repository details, editing scope, and media paths.

AppConfig#

Each project is configured through the setup form. The configuration is stored server-side in SQLite for logged-in users, or in localStorage for anonymous sessions.

Setting Type Description
websiteUrl string The live URL of the site (used to fetch HTML and resolve relative paths)
githubRepo string Repository in owner/repo format
filePath string Path to the HTML file within the repo
githubToken string Personal access token (encrypted with AES-256-GCM in the database)
mediaFolder string Path to the media/images folder in the repo
assetsFolder string Path to additional assets (CSS, JS files)
editableSelectors string Custom CSS selectors defining editable scope
sectionsSelector string Custom selector for section discovery
cssFile string Path to a CSS file for live editing
jsFile string Path to a JS file for live editing

Phase-Based App Flow#

The app uses a state machine: setup → loading → editing, plus dashboard for logged-in users with projects.

  • Setup — shows the setup form, reads saved config from localStorage
  • Loading — fetches HTML (from URL or GitHub repo)
  • Editing — visual editor with structured panel and toolbar
  • Dashboard — project list for logged-in users (at /dashboard)

If a valid saved config exists in localStorage, the app skips setup and goes directly to loading on page refresh.

Custom Selectors

Control which elements are editable using scope selectors, container syntax, and data attributes.

Editable Selectors#

By default, Klar makes all standard text elements editable across the entire page. The editableSelectors setting lets you restrict which areas of the page are editable by providing CSS selectors that define the scope.

Selectors 
# Only elements inside .hero and .content are editable
.hero, .content

# Editable except for links and images inside .hero
.hero:not(a, img), .content

# Mark a container for list editing
.feature-list[]

# Editable background image
.hero__bg:style(background-image)

Selector Syntax#

Syntax Meaning
.hero All editable elements inside .hero
.hero:not(a, img) Everything in .hero except links and images
.feature-list[] Treat .feature-list as a container (list editing)
.bg:style(background-image) Enable background-image editing on .bg

Data Attributes#

You can also mark elements directly in your HTML:

  • data-klar-edit or data-klar — mark any element as editable regardless of tag name
  • data-klar-list or data-list — mark a parent as a list container
  • data-klar-style-bg — enable background-image editing

Architecture Overview

Key architectural patterns and data flow in Klar.

Data Flow#

Props-down, callbacks-up — no Context API or external state libraries. App.tsx owns all top-level state (auth, config, edits, HTML). EditorLayout is the coordination hub that wires VisualEditor, StructuredPanel, and Toolbar together.

Centralized Edit State#

All edit tracking flows through a single useReducer in useEditTracker.ts. Edits are keyed by CSS selector (optionally suffixed with ::attribute). When adding an edit, the reducer preserves the first original value — subsequent edits update edited but keep the initial original. A useRef keeps a synchronized snapshot for save operations.

Full-Stack Data Pattern#

Adding a new project setting follows this path through the stack:

  1. DB column in server/db.js — ALTER TABLE ADD COLUMN in try/catch for idempotency
  2. Server mapper in server/db.js — snake_case DB → camelCase JS in query results
  3. API route in server/index.js — destructure from request body, pass to upsertProject
  4. TypeScript types — Project in src/lib/auth.ts, AppConfig in src/types.ts
  5. React state — wire through App.tsx → SetupForm.tsx → consumer components

Local Persistence#

UI preferences (viewport size, panel open state) persist via localStorage using a thin wrapper in src/lib/settings.ts. Setup form values also persist to localStorage for auto-start on reload.
